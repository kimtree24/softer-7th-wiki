## 2. 1일 리뷰
팀 미션 진행한 것을 바탕으로 피드백을 받았다.
강사님께서 최소비용-최대효과는 잘못된 정의라고 코멘트 달아주신 것에 대해 혼자 생각을 해보았을 때, 우리 팀에서 선정한 결혼 준비 데이터 제공 플랫폼에서의 문제 정의가 잘못되었다는 판단을 했다.
최소 비용, 최소 시간, 최대 효과를 고려했을 때 신혼 부부는 최대 효과를 선택할 것이고, 그렇다면 우리가 문제 해결을 위해 제시한 솔루션은 효용, 비용과 관련된 것이기에 적절하지 않다고 판단했다.
나아가 최대 효과를 수요자가 원한다면 우리 서비스를 사용하기 보다는 웨딩 플래너를 이용하거나, 애당초 문제가 되지 않을 수도 있다고 판단했다.
만약에 이러한 주제를 계속 이어간다면, 오히려 수요자를 신혼 부부가 아니라 웨딩 플래너로 타겟을 변경하여 웨딩 플래너가 여러 업체 정보를 원활하게 수집하고, 이를 이용할 수 있도록 하는 것이 더 좋은 방향일 것이라는 결론에 다다랐다.

이번 주 미션을 수행하면서 가장 큰 고민은 짧은 기간 동안 문제 정의도 설득력 있게 하고, 개인 미션도 수행하며, 팀 프로덕트도 제출해야 한다는 시간적 압박이 심했던 것 같다.

우리 팀은 팀 미션 수행하며 이미 프로토타입 구성 단계에서 피보팅도 진행했기에 시간이 절대적으로 부족했다.

강사님께 이러한 질문을 드렸을 때 강사님께서는 아이디에이션 단계에서 더 많은 피드백을 주고 받으며 그 주기를 짧게 가져갔어야 한다고 답변을 주셨다.
더불어 이런 리스크 관리에 대해 다음 주차에서 설명해 주신다고 하셨다.

문제 정의의 중요성은 인지하겠으나, 엔지니어링쪽으로 더 신경을 쓰고 싶은 마음이 있기에 이 두가지를 어떻게 밸런싱을 잘 할 것인지가 앞으로 이 코스를 수행하며 계속 생각해봐야 할 부분이 아닌가 한다.

## 4. 오늘의 학습

### (1) 예외 처리(try / except)의 올바른 사용 원칙

- 예외 처리는 “내가 예상하는 실패 상황”을 명확히 정의 / 해당 상황이 발생하는 코드만 try 안에 배치 / 그 실패 상황에 대응하는 except를 지정
- try 안에 큰 코드 덩어리를 넣으면 어떤 줄에서 예외가 발생했는지 알기 어렵고 / 내가 의도한 예외인지, 전혀 다른 버그인지 구분할 수 없다.
- 실패할 수 있다고 생각한 코드만 try로 감싸고, 그 실패에만 except로 대응
- 예외 처리는 에러를 띄우는게 아니라, 의도한 실패 시나리오를 구조화하는 것임

### (2) 샘플링(Sampling)의 필요성과 전략

#### 1) 왜 샘플링이 필요한가
- 현실의 데이터는 너무 커서 전체 데이터를 다 분석하려 하면 시간, 비용 많이 소요됨
- 초기 단계에서는 작은 데이터로 POC(Proof of Concept)를 만들고, 사업성이 보이면 확장하는 전략이 필요하다.

#### 2) 샘플링은 단순 추출이 아니다
샘플링에는 반드시 의도가 들어가야 한다.
```
	•	그냥 무작위 추출 X
	•	문제를 잘 드러내는 하위 집합(subset) 선택 O
```

즉, 샘플링은 큐레이션의 시작점.

#### 3) 올바른 샘플링 구조
```
	1.	먼저 전체 데이터에서
	2.	긍정 / 부정을 구분한 뒤
	3.	각 그룹 내부에서 샘플링
```

잘못된 방식
```
전체에서 샘플 → 그 안에서 긍/부정 분리
이 방식은 특정 감정이 과소/과대 대표될 위험이 크다.
```

### (3) 문제 정의 타겟을 좁히는 이유

- 고통이 큰 사람일수록 문제를 정확하게 설명해 주고, 솔직한 피드백을 주며, 솔루션의 방향을 빠르게 교정해 준다
- 즉, 타겟을 좁히는 것은 판매 전략이 아니라 문제 정의 전략이다.

### (4) 시각화 도구 선택의 원칙

- 어떤 시각화 툴을 쓸지 고민할 때, 도구부터 고르면 안 된다.

```
	1.	Use case (무엇을 알고 싶은가)
	2.	가설 (이 데이터에서 무엇이 보일 것이라 예상하는가)
```

#### 1) 시각화는 검증 도구다

올바른 흐름은 다음과 같다.

```
	1.	문제를 정의하고
	2.	가설을 세운 뒤
	3.	데이터를 의도적으로 샘플링하고
	4.	그 결과를 시각화해서
	5.	실제로 그렇게 보이는지 확인한다
```

즉, 시각화는 장식이 아니라 가설 검증 장치이다.


#### 2) 프로토타입의 역할

초기 단계에서 중요한 것은 빠르고, 작고, 단순한 시각화

```
	•	완벽한 시스템 X
	•	빠르게 만들어서 보여주고, 반응을 확인
```

이를 통해

```
	•	문제 정의가 맞는지
	•	데이터가 의미 있는지
	•	다음 단계로 갈 가치가 있는지
```

를 빠르게 판단할 수 있다.
